# Three-Turn Architecture  A compression of architectural decision-making into three user-facing turns: **Survey â†’ Spec â†’ Skeleton+Slice**.  The sibling of `three-turn-design`. Where the design skill compresses pixel-level forks, this skill compresses boundary, contract, and failure-mode forks. The artifact is not a mockup â€” it is an executable contract (interface signatures + sequence diagram + failure-mode table) that the next turn implements.  The skill assumes the user has declared invariants once and now wants execution, not debate. It refuses to add a fourth turn â€” if a fourth would be needed, the skill chooses a default rooted in the invariants, parks the question, and proceeds.  ## When to invoke  - The user names an architectural constraint as a problem ("this is too coupled", "p99 is climbing", "every deploy needs all three services up", "one bad payload kills the queue") and wants a fix shipped. - The user signals speed: "quickly", "rough cut", "skeleton it", "give me the bones", "I don't want to debate every contract". - A `~/.claude/architecture-defaults.md` (or per-project equivalent) exists and is recent. - The codebase is readable â€” the skill can ground itself in actual topology (imports, calls, data ownership) before designing.  ## When to refuse  - (No refuse rule for "no defaults file" â€” handled inline; see Turn 1 setup below.) - The work is exploratory ("how should we architect this product?"). Route to `superpowers:brainstorming`. - The user wants to weigh in on each boundary call. Three-turn-architecture is wrong for that mode. - The change is one component deep with no boundary crossed. That's an implementation task, not an architecture task â€” just do it. - The prompt is a vendor / tool / model selection ("which API should I use?", "which library?", "which Hugging Face Space?", "which LLM?"). That's a procurement decision, not an architectural one. Make a direct recommendation outside the three-turn frame â€” do not invent contracts, sketches, or slices for an external service the user doesn't own.  ## The three turns  ### Turn 1 â€” Survey & single architectural fork  Before responding, the skill: 0. **Ensures `architecture-defaults.md` exists** at a canonical path (`~/.claude/architecture-defaults.md` home, or `<project>/.claude/architecture-defaults.md` per-project; per-project overrides home). If no defaults file exists, the skill **scaffolds one inline and proceeds** in the same response â€” this preserves the three-turn budget. **Scaffold guardrails:**    - Write to the canonical path. NEVER write to the project root or any non-canonical location.    - Pre-fill aggressively from project inference (read CLAUDE.md, package.json, framework hints, existing CSS animation durations, etc.) â€” the user's review should be "skim and tweak," not "fill out 12 fields."    - In the same response that scaffolds, surface (a) the scaffold path and (b) a one-line audit summary of the inferred invariants (e.g., "scaffolded with: timeout 800ms inferred from CSS keyframe, in-proc boundaries, flag-gate-required, idempotency-required").    - State explicitly that "push back" at the Turn 1 fork question covers "defaults are wrong, let me edit them" â€” the user can override the agent-inferred invariants by pushing back, the defaults file, or both. 1. Reads the actual topology as ground truth â€” `grep` the call sites, read the data-ownership boundaries, inspect the existing contracts. **No assumption-based architecture.** 2. Reads `architecture-defaults.md` and any project-specific overrides. 3. Names the actual constraint in **one sentence** (latency? coupling? blast-radius? deploy-independence? data-ownership? versioning?). 4. Renders **two** architectural sketches as **executable artifacts** via the architecture companion (see `companion/README.md`):    - Interface signatures (typed, not pseudo)    - Sequence diagram (mermaid, one happy path + one failure path)    - Failure-mode table (timeout / retry / idempotency / fallback per cross-boundary call)    - Perf envelope (latency p50/p99 budget, throughput target, memory ceiling)     Invocation: write the sketch JSON to `/tmp/three-turn-arch-turn1.json`, then run    `python3 ~/.claude/skills/three-turn-architecture/companion/render.py /tmp/three-turn-arch-turn1.json --open`    and report the resulting `file://` URL alongside the recommendation.     **Inline fallback (only when the browser is genuinely unavailable):** print a markdown analogue with the *same four headed sections per sketch* â€” `### Interfaces`, `### Sequence (happy + failure)`, `### Failure modes`, `### Perf envelope` â€” not free-form prose. State the reason for falling back ("companion render failed", "no display available") in one sentence. Free-form prose is NOT an acceptable substitute.     **In-process boundaries:** when the standard failure-mode columns (timeout / retry / idempotency / dead_letter) don't literally apply because the call is in-process (React state, function-to-function, in-memory queue), the table is still required. Use `n/a (in-process)` in the columns that don't apply, and add a row for each in-process failure mode that DOES apply â€” memory leak, frame jank, state desync, event-loop starvation, dropped event under backpressure. Same for perf envelope: replace network-shaped metrics with the relevant in-process budget (frame budget for 60fps = 16.7ms p99, allocation rate, listener count). 5. States a recommendation with a one-line rationale rooted in a defaults invariant. 6. Asks **one** question: "A, B, or push back?"  Bundled into the recommendation (not surfaced as separate questions): - Sync vs async transport - Persistence boundary (which side of the cut owns state) - Versioning / contract evolution policy - Observability hooks (metric names, log shape, trace points) - Failure-mode coverage (timeouts, retries, idempotency keys, dead-letter) - Rollback path  Forbidden: - 3-way or 4-way options - Sub-questions ("sync or async? then which queue? then retention?") - Asking "should I sketch this?" â€” just render - Boxes-and-lines diagrams without typed contracts underneath - "Microservice vs monolith" framing without naming the actual cut and its data ownership  ### Turn 2 â€” Spec & contracts  After the user picks (or pushes back once, which counts as Turn 1 redux):  1. Apply all preference defaults silently. 2. Decide the slice scope before writing:    - If the first vertical slice fits in **one boundary cross** (one route â†’ one queue â†’ one handler, or one read path end-to-end), plan B (skeleton + slice).    - If the slice would cross multiple boundaries or touch shared state across cuts, **auto-degrade to A** (skeleton-only) and list the slice as ticket #1. 3. Render the consolidated architecture via the companion (`turn: 2` JSON with the `consolidated` block containing migration plan, slice tickets, and parked items). Invocation:    `python3 ~/.claude/skills/three-turn-architecture/companion/render.py /tmp/three-turn-arch-turn2.json --open`    Components, typed interfaces, data flow, failure modes, perf envelope, migration path (parallel-run window + flag name + cutover trigger), slice plan all live in this single rendered view. 4. Write the spec to `docs/architecture/specs/YYYY-MM-DD-<topic>-arch.md`. 5. Park 5â€“10 sub-decisions as tickets in the spec (cache eviction policy, exact retry curves, dashboard layouts, log field names, alert thresholds). 6. Ask **one** question â€” the **4-option commitment ladder**: **"Ship slice / Ship skeleton, defer slice / Ship spec, defer code / Tweak first?"**     | Choice | What ships | Commitment level |    |---|---|---|    | Ship slice | Spec + skeleton + one vertical thread end-to-end | â–®â–®â–®â–® Most |    | Ship skeleton, defer slice | Spec + typed stubs + no-op handlers (types compile, nothing runs) | â–®â–®â–® |    | Ship spec, defer code | Spec doc only, zero code touched | â–®â–® |    | Tweak first | Modify spec/sketch, re-evaluate before committing | â–® Pre-commit |     The ladder makes the commitment level explicit so the user does not invent off-menu hybrids (Run 03 dogfood evidence: user replied "A x B" to the old 3-option menu because they didn't see that slice âŠƒ skeleton). "Push back / redesign the fork" is covered as free-text under "Tweak first."  Forbidden: - Showing 2+ architectures again (we already forked) - Asking about parked items - Reopening a settled boundary decision - "Should I write the spec?" â€” just write it - A migration plan without a rollback path  ### Turn 3 â€” Skeleton + first vertical slice  After the user says "ship" / "ship slice" / "skeleton only" / names one tweak:  1. Implement the skeleton: interfaces, types, contract stubs, migration scripts, alert/metric registration. 2. If B (slice): implement the vertical slice end-to-end through the new architecture. 3. Run the type check. 4. Run contract tests at every new boundary. **A contract test exercises the actual logic of the new boundary** â€” for an in-process boundary (e.g., a hook's queue â†’ consumer chain), this means triggering the producer and asserting the consumer fires in the expected order with the expected payload, not a `curl` against the served HTML. For a network boundary, it means hitting the new endpoint and validating the response shape. A "the page loads" smoke is not a contract test; it is a liveness check at best. 5. Run the build. 6. Run a smoke test against the slice (if B) or against the boundary stubs (if A). The smoke test is **separate from and additional to** the contract test in step 4. Smoke = "does the deployed thing respond at all"; contract = "does the new boundary's logic behave correctly." 7. Restart the affected service (if applicable). 8. Report: files touched (count + key paths), interfaces landed (signatures), slice status (passing / skeleton-only), parked tickets (numbered list), and **`build_count`** â€” how many builds it took to ship clean. For each build > 1, also note **what broke** and **whether it matched the Turn 1 typed interface** (Y = implementation mistake; N = Turn 1 rigor failure).  **Build-count guardrail.** `build_count` is a leading indicator of Turn 1 rigor â€” clean types in Turn 1 produce clean builds in Turn 3. Thresholds: - **1 build** â€” clean. Turn 1 types were committable verbatim. - **2 builds** â€” soft drift. One typed-interface iteration; log in `observation-log.md` and continue. - **3+ builds** â€” hard drift. Turn 1 rigor failed materially. **Stop after the failing build.** Surface a one-line diagnosis ("Turn 1 declared X; implementation needed Y") before asking for the next instruction. Do not silently keep iterating. - The per-project ceiling is configurable via `max_builds_per_slice` in `architecture-defaults.md` (default: 2). When exceeded, the hard-drift handling above kicks in regardless of literal count.  Forbidden: - Pausing for "did the slice work?" verification before running smoke - Asking permission to rebuild, restart, or run migrations - Surfacing a parked item as a new question - Adding scope ("while I was in there I also fixed the unrelated thing") - Shipping a migration without its rollback in the same commit  ## Compression rules  The skill is allowed exactly **three user-facing turns** between "start" and "ship." If a fourth would be required:  - Choose the most defensible default, rooted in an `architecture-defaults.md` invariant. - Document it as a parked ticket in the spec with a one-line rationale and the defaults clause that drove it. - Proceed to the next turn.  Bundled decisions, not menus. Recommendations, not options. Defaults, not questions. **Contracts before implementations** is the load-bearing invariant.  ## Auto-degrade from B to A  The slice exists to prove the architecture works under one realistic request. The slice is too big when:  - It crosses more than one new boundary. - It depends on a migration that hasn't finished its parallel-run window. - It requires data fixtures that don't exist yet. - The user's request implicitly bundles two slices ("the read path AND the write path").  When any of these hold, **auto-degrade silently** â€” write the slice as ticket #1 in the spec, ship the skeleton, and report the degrade in the Turn 3 summary ("Shipped skeleton; slice listed as ticket #1 because <reason>"). Do NOT use this as a fourth turn.  ## Required: `architecture-defaults.md`  The skill reads this on entry. Without it, the skill cannot be opinionated enough and the compression fails. See `architecture-defaults-template.md` in this skill folder for the format.  The file declares (at minimum): - **Boundary policy** â€” what defines a service / module / package cut - **Data ownership** â€” no shared mutable state across cuts; who owns persistence - **Failure-mode discipline** â€” timeout, retry, idempotency, dead-letter defaults for every cross-boundary call - **Performance envelope** â€” latency budgets (p50, p99) per tier; memory ceilings; throughput targets - **Observability budget** â€” every boundary emits â‰¥1 metric + 1 structured log; trace points declared - **Versioning policy** â€” additive evolution; breaking changes need migration plan + parallel-run - **Migration cadence** â€” parallel-run window length, feature flag naming, cutover trigger, rollback path - **Testing discipline** â€” contract test per new boundary; smoke test per slice; load test threshold for perf-sensitive paths - **State-mutation policy** â€” API routes vs direct DB (matches `three-turn-design`'s rule) - **Forbidden patterns** â€” e.g., shared databases across services, sync chains >2 hops, hot-loops in request path, irreversible writes alongside schema changes  ## Why this works  In sessions this skill is meant to replace, roughly one-third of user turns were: - Invariant inputs (timeout defaults, observability rules, migration cadence) that could be pre-declared. - Verification corrections ("you didn't add the rollback", "no metric on that boundary", "where's the idempotency key"). - Boundary-scope debates that resolved by reading the actual call graph rather than discussing.  Eliminate those buckets and the remaining architectural taste calls collapse into 2â€“3 bundled decisions. Net: ~3 user-facing turns to a working skeleton with one slice through it.  The compression formula: > **Binding invariants + grounded survey + single-fork-per-turn + skeleton-before-impl** > consultative iteration.  ## Testing  See `tests/README.md` for how to run scenarios and log observed agent behavior.
