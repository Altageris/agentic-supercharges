# BFS  Explains a topic in discrete, interruptible steps. Each section is delivered as its own turn with a brief prompt â€” the user can cut in at any point to redirect, ask a follow-up, or stop. Agents can invoke this skill to explain intermediate state to the user during long-running work.  **Arguments:** - The topic, question, code snippet, or text to explain (e.g. `/bfs how does ActionLegality.py filter moves?`) - `--agent` flag â€” suppress the continue-prompts and emit all sections in sequence without pausing (for use when called from within an agent). **Default behavior when invoked by agents** to avoid continuation context loss. - `--sections N,M,K` flag â€” deliver only specific sections (e.g., `--sections 1,5` for core idea + mental model). Reduces token cost for agents that don't need all sections. Only meaningful with `--agent`. - `--agent-interactive` flag â€” enable interactive mode for agents with automatic state tracking across SendMessage continuations. Agent can respond "Continue" and bfs will auto-advance to the next section, or redirect with "jump to X". (Not recommended for large-scale testing due to agent continuation overhead.) - `--feedback "<comment>"` flag â€” **does not deliver an explanation**. Instead, appends the comment to `FEEDBACK.md` next to this skill and exits. Used by the human partner to record what worked, what didn't, or what to try differently next time. `skill-audit` aggregates these entries during audit runs.  ## Sections (BFS levels)  Deliver these in order, one per turn:  1. **Core idea** â€” one sentence, no jargon 2. **Why it exists** â€” the problem it solves (1â€“2 sentences) 3. **How it works** â€” the mechanism in plain terms (short paragraph or steps) 4. **Key parts** â€” named sub-components or cases, one line each 5. **Mental model** â€” a single analogy or phrase the user can hold onto  ## Steps  ### Parse arguments - Extract the topic from ARGS - Check if `--agent` flag is present - Check if `--agent-interactive` flag is present - Check if `--feedback` flag is present â€” if yes, go to **Feedback mode** below and do not deliver an explanation - **Auto-detect agent context** â€” if no flags are explicitly set, but this skill is being invoked as part of an agent's work (detected via execution context):   - If `interactiveSkillState.bfs.enabled` is true in settings.json: default to interactive mode (which will use state persistence)   - Otherwise, default to `--agent` (batch mode) to avoid continuation overhead  ### Pre-flight: read accumulated feedback Before emitting **Section 1**, check for `FEEDBACK.md` next to this `SKILL.md`:  ```bash ls /Users/AI-CCORE/.claude/skills/bfs/FEEDBACK.md 2>/dev/null && \   cat /Users/AI-CCORE/.claude/skills/bfs/FEEDBACK.md ```  If entries exist, treat them as guidance for **this call**: - Reshape section wording, length, or emphasis to reflect the notes - If an entry says a section is often unhelpful for a given topic type, consider compressing or skipping it - Do not surface the feedback to the user â€” it is Claude's to apply, not to narrate  Feedback entries are guidance, not hard rules; they override defaults but lose to explicit user instructions in the current call.  ### Interactive mode (default â€” no `--agent`) Deliver **Section 1** (Core idea) first. End with: > *"Continue? Or jump to a specific part â€” why it exists / how it works / key parts / mental model."*  Wait for user input before proceeding.  On each subsequent turn: - Deliver the next section - End with the same brief prompt (omit after the final section) - If the user redirects ("jump to key parts", "go deeper on X"), honor it immediately â€” skip ahead or expand that branch  The user interrupting at any turn is expected and correct. No section is mandatory.  ### Agent mode (`--agent`) Skip all continue-prompts. Emit requested sections in a single response, clearly labeled. The invoking agent controls the flow â€” do not pause for user input.  **Default for agents:** When an agent invokes bfs without explicit flags, default to `--agent` mode (batch delivery). This avoids SendMessage continuation overhead and context loss.  **Cost optimization with `--sections`:** Agents can specify which sections to deliver via `--sections N,M,K`: - `--agent --sections 1` â€” core idea only (minimal cost) - `--agent --sections 1,5` â€” core idea + mental model (common for quick validation) - `--agent --sections 1,3,4` â€” core idea + how it works + key parts (mechanism focus) - `--agent --sections 1,2,3,4,5` (or omit `--sections`) â€” all sections (full explanation)  **Example agent invocations:** ``` # Full explanation, batch mode /bfs --agent "Explain X"  # Just core idea and mental model (saves tokens) /bfs --agent --sections 1,5 "Explain X"  # Just the mechanism (useful for understanding) /bfs --agent --sections 3 "Explain X" ```  If `--sections` is used without `--agent`, the flag is ignored (interactive mode doesn't support partial delivery).  ### Agent-interactive mode (`--agent-interactive`) **Enables interactive continuation for agents without multi-spawn context loss.**  When invoked with `--agent-interactive`, bfs works with Codex's `interactiveSkillState` settings to maintain state across agent continuations:  1. Deliver Section 1 with the continue-prompt: *"Continue? Or jump to a specific part â€” why it exists / how it works / key parts / mental model."* 2. **Save state to Codex context** â€” store section counter (1â€“5), topic, and any redirects in the session state (managed by settings.json `interactiveSkillState.bfs`) 3. **On agent continuation** â€” when the agent responds via SendMessage with "Continue" or a redirect ("jump to X"):    - Codex retrieves saved state (section, topic, redirect)    - Recognize the response as a valid continuation (not a new invocation)    - Auto-advance to the next section or jump to the requested section    - Deliver that section with the continue-prompt    - Update state and repeat until Section 5 (omit prompt after final section) 4. **Preserve context across invocations** â€” state persists at the platform level, so each agent continuation knows which section was last delivered  **Configuration (in ~/.claude/settings.json):** ```json "interactiveSkillState": {   "enabled": true,   "persistenceMode": "memory",   "skills": {     "bfs": {       "enabled": true,       "stateFields": ["section", "topic", "redirect"],       "description": "Track section number, topic, and redirection across agent continuations"     }   },   "agentContinuationTimeout": 600 } ```  **Example agent flow:** ``` Agent invokes:  /bfs --agent-interactive "Explain concept X" BFS delivers:   Section 1 + continue-prompt              (State saved: section=1, topic="Explain concept X") Agent responds: "Continue"              (Codex retrieves state, passes to bfs) BFS delivers:   Section 2 + continue-prompt              (State updated: section=2) Agent responds: "Jump to mental model"              (Codex retrieves state, passes redirect to bfs) BFS delivers:   Section 5 (mental model) ```  This mode is ideal for agent-driven testing and validation scenarios where a single agent needs to step through all 5 sections interactively without spawning multiple agents. Works with scenarios 02 (redirection) and 05 (compression) without context loss.  ### Feedback mode (`--feedback "<comment>"`)  **Implementation:** This mode must be invoked by Claude directly (not via agents, which lack file write permission).  When Claude detects the `--feedback` flag: 1. Extract the comment from the argument (verbatim, no paraphrasing) 2. Get today's date (YYYY-MM-DD format) 3. Read `/Users/AI-CCORE/.claude/skills/bfs/FEEDBACK.md` 4. Check if today's date heading exists (e.g., `## 2026-05-13`) 5. If heading exists: append `- <comment>` under it 6. If heading does not exist: create `## YYYY-MM-DD` heading, then append `- <comment>` on next line 7. Write the updated file back 8. Return a one-liner confirmation: "recorded." 9. Do not deliver any explanation or sections  **Behavior:** - One comment per invocation, recorded verbatim - No paraphrasing, summarisation, or editorialisation - Stop immediately after writing  **File path:** `/Users/AI-CCORE/.claude/skills/bfs/FEEDBACK.md`  The file is read by `skill-audit` during audit passes (so accumulated feedback shows up in the report), and by this skill's pre-flight step (so guidance applies to the next explanation).  ---  ## State Tracking for Agent-Interactive Mode  **Problem solved:** Previously, agents invoking bfs interactively would lose context across SendMessage continuations, requiring multiple agent spawns and causing context fragmentation.  **Solution: Built-in session state**  When `--agent-interactive` is active: 1. Maintain an internal session context that tracks:    - Current section number (1â€“5)    - Topic/prompt string    - Whether a redirect was requested (e.g., "jump to key parts") 2. Store this state in memory during the skill execution 3. On each continuation request from the agent, retrieve the state and proceed to the next section 4. No multi-agent spawns needed â€” one agent handles all 5 sections  **Key invariants:** - Only agents (or explicitly-flagged interactive sessions) use this mode - State is session-local and does not persist across separate skill invocations - Redirection ("jump to X") skips intermediate sections and delivers the requested one - Final section (5) omits the continue-prompt  This eliminates the context-loss issue and allows agents to step through BFS explanations in a natural, linear flow.
